# Configuration file

The default configuration file is `sftpgo.json`. It is divided into the following sections.

## Common

Configuration parameters for the `common` section.

| ----------- | ------ | --------- | ------------- |
| `idle_timeout` | integer | `15` | Time in minutes after which an idle client will be disconnected. `0` means disabled. |
| `upload_mode` | integer | `0` | Upload behavior flags. Can be combined (see below). Ignored for the SFTP backend if buffering is enabled. |
| `setstat_mode` | integer | `0` | `0`: normal mode (permission/owner/time change requests are executed). `1`: ignore mode (requests are silently ignored). `2`: ignore if not supported (permission/owner changes are silently ignored for cloud filesystems, executed for local/SFTP). |
| `rename_mode` | integer | `0` | `0`: renaming non-empty directories is not allowed for cloud storage providers (S3, GCS, Azure Blob). `1`: enable recursive renames for these providers. Recursive renames may be slow and are not atomic; partial renames and incorrect quota updates are possible on error. |
| `symlink_mode` | integer | `0` | Bit mask selecting the backends on which clients holding the `create_symlinks` permission may create symbolic links. `0` (default): creation disabled on every backend. `1`: allow on the local filesystem, including its encrypted variant. `2`: allow on the SFTP backend. `3`: both. See [Symbolic links and permissions](#symbolic-links-and-permissions). |
| `resume_max_size` | integer | `0` | Maximum file size in bytes for which upload resume is allowed on storage backends with immutable objects (S3, GCS, Azure Blob). SFTPGo must rewrite the entire file on resume. `0` means resume is disabled. |
| `secret_min_entropy` | float | `80` | Minimum entropy required for plain-text data-at-rest encryption secrets: the CryptFs passphrase and the S3 SSE-C key. These secrets are key material and must be random, not a memorable password. The value is a heuristic estimate (character-set and length based, like password-strength estimators) that screens out obviously weak secrets: clearing the check confirms the secret is not trivially guessable, not that it is cryptographically strong. Treat it as a safety net, and rely on always supplying key material generated by a CSPRNG for the actual strength. The check runs when the secret is submitted in plain text (create, change, REST API, Terraform, `loaddata`), so secrets that are already stored keep working when the threshold is raised. | `umask` | string | empty | File mode creation mask (e.g., `002`). Leave blank to use the system default. Supported on *NIX platforms only. |
| `server_version` | string | empty | Customize the advertised software version. `short` hides the version number. The commit hash may still be appended; this value may be ignored when SFTPGo acts as a client (e.g., for S3). |
| `tz` | string | empty | Time zone for the EventManager scheduler and time-based access restrictions. Set to `local` for server local time; otherwise UTC is used. |

### Symbolic links and permissions

Symbolic links exist on the local filesystem and SFTP backends; cloud object storage (S3, Azure Blob, Google Cloud Storage) has no symbolic links, so this section applies to those two backends. Symbolic links are resolved before each operation and confined to the user's accessible area — the home directory for the local backend, the configured prefix for the SFTP backend. A link that resolves outside it is rejected rather than followed. For the SFTP backend this is enforced by resolving paths on the remote server and so assumes the remote is trusted; see [SFTP root directory and path confinement](sftpfs.md#sftp-root-directory-and-path-confinement) for what the prefix guarantees on a remote you do not control.

Within that area, on these backends, clients holding the `create_symlinks` permission can create symbolic links when `symlink_mode` enables creation for the backend; it is disabled by default (see below). Creation also requires that permission on both the link's directory and the directory the link points into. Most file operations **dereference** a symbolic link: reading, writing, and metadata changes act on the link's *target*, not on the link node. Deleting or renaming a link, by contrast, acts on the link node itself, not on its target. Per-directory permissions are evaluated at the path the client requests, so a link placed in one directory that points into another runs operations under the permissions of the link's directory while acting on the target. When the two directories are governed by different per-directory permissions, operating through the link can reach the target in ways the target's own directory would not allow.

Dereferencing a link never re-checks `create_symlinks`: any link already present on the storage is followed under the requesting path's permissions. Only *creation* through SFTPGo is gated, and because it requires `create_symlinks` on the directory the link points into, the permission is **bidirectional** — granting it on a directory authorizes that directory both as a place to create links and as a place links may point into. A directory that carries `create_symlinks` can therefore be made a link target from any other directory that also carries it, reached under that other directory's permissions. A directory that does **not** grant it cannot be made a *new* link target through SFTPGo, but this does not retract links that already exist: one created while the permission was granted, or placed on the storage outside SFTPGo, stays followed regardless. Do not grant `create_symlinks` on a directory whose download, upload, or overwrite permissions you restrict; and because revoking it later leaves existing links in place, keep creation disabled (the default `symlink_mode`) for any backend whose clients do not need it.

`symlink_mode` is a bit mask that selects the backends on which clients may create symbolic links, regardless of the `create_symlinks` permission. The default, `0`, disables creation on every backend across all protocols; add `1` to allow it on the local filesystem (including its encrypted variant), `2` on the SFTP backend, `3` for both. Symbolic links already present on the storage are still honored: `symlink_mode` controls creation, it does not change links that already exist.

:information_source: Confinement validates each path before the operation runs, so a symbolic link resolving outside the accessible area is rejected and a persistent link can never reach beyond it. Because validation and the operation are separate steps, a brief timing window remains under highly concurrent access: a crafted link could be momentarily followed between the check and the operation. Symbolic-link creation is disabled by default, so a client cannot introduce a link to exploit this; for environments with strict isolation requirements, provision each user from an empty directory that contains no symbolic links, so that with none present and none creatable there is nothing to act on.

:warning: Per-directory permissions — and every other path-based restriction, such as file pattern filters and allowed/denied share paths — control access *through SFTPGo* and assume the underlying storage layout is trusted. They are evaluated against the path the client requests, not the location it resolves to: if the storage redirects one path to another (symbolic link, bind mount, hard link, Windows junction), SFTPGo applies the restrictions configured for the requested path, even when the resolved target is configured more strictly. Such redirections placed on the storage outside SFTPGo (directly on the filesystem or by another tool) can therefore cross restriction boundaries, and SFTPGo cannot always detect them. If you rely on these restrictions to isolate parts of a user's accessible tree, provision the storage so its directory structure reflects the boundaries you intend; symbolic-link creation is disabled by default, so leave it disabled when clients do not need links. See [How path-based restrictions are evaluated](access-control.md#how-path-based-restrictions-are-evaluated).

### `upload_mode` flags

| Flag | Description |
| ------ | ------------- |
| `0` | **Standard**: files are uploaded directly to the requested path. |
| `1` | **Atomic**: files are uploaded to a temporary path and renamed on success. On error, the temporary file is deleted. Avoids serving partial files. |
| `2` | **Atomic with resume**: same as atomic, but on error the temporary file is renamed to the requested path (not deleted), allowing clients to resume. If both `1` and `2` are provided, `2` takes precedence. |
| `4` | Store partial files for the S3 backend even if a client-side upload error is detected. |
| `8` | Store partial files for the Google Cloud Storage backend even if a client-side upload error is detected. |
| `16` | Store partial files for the Azure Blob backend even if a client-side upload error is detected. |

### `proxy_protocol` modes

| Mode | Description |
| ------ | ------------- |
| `0` | Disabled. |
| `1` | Enabled. The proxy header is always read. If the upstream IP is not in `proxy_allowed`, the header is ignored (but the connection is accepted). Use `proxy_skipped` to allow specific IPs/networks to connect without sending or having their proxy header read. |
| `2` | Required. The proxy header is always read. If the upstream IP is not in `proxy_allowed` and a proxy header is found, the connection is rejected. Use `proxy_skipped` to allow specific IPs/networks to connect without sending or having their proxy header read. |

| Parameter | Type | Default | Description |
| ----------- | ------ | --------- | ------------- |

| `entries_soft_limit` | integer | `100` | Ignored for the `provider` driver. |
| `entries_hard_limit` | integer | `150` | For the `memory` driver, the number of banned IPs and host scores kept in memory varies between the soft and hard limits. For the `provider` driver, this limits the number of entries returned when listing all defender hosts. |

## ACME

| `email` | string | empty | Email used for registration and recovery contact. |
| `key_type` | string | `4096` | Key type for private keys. Supported values: `2048` (RSA 2048), `3072` (RSA 3072), `4096` (RSA 4096), `8192` (RSA 8192), `P256` (EC 256), `P384` (EC 384). |

| `renew_days` | integer | `30` | Number of days before expiration at which to renew the certificate. |

### node

:information_source: Here a *node* is a single running SFTPGo instance (process), identified by the `host`/`port` below. On Kubernetes this is a **pod**, not a Kubernetes worker node — several SFTPGo pods can run on the same Kubernetes node, and each counts as a separate SFTPGo node.

:information_source: Inter-node communication provides a uniform *view* of the cluster; it does not pool *enforcement*. Resource and concurrency limits — such as per-user `max_sessions`, bandwidth limits, the per-host and total connection caps, and rate limits — are scoped to a single node: each node enforces them independently against the traffic it serves and does not coordinate them with other nodes, unless a setting is explicitly documented as cluster-wide. Plan capacity accordingly: a per-node limit of N can allow up to N times the number of nodes across the cluster.

## KMS

Configuration parameters for the `kms` section.

### secrets

| Parameter | Type | Default | Description |
| ----------- | ------ | --------- | ------------- |
| `url` | string | empty | URI to the KMS service. |
| `master_key` | string | empty | Active master encryption key as a string, used to encrypt and as the first key tried on decryption. |
| `master_key_path` | string | empty | Absolute path to a file containing the active master encryption key. If not empty, takes precedence over `master_key`. |
| `additional_master_keys` | list of strings | empty | Decrypt-only master keys, tried in order only when the active key fails to decrypt a secret. They never encrypt. See [Master key ring](#master-key-ring). |
| `additional_master_keys_path` | string | empty | Absolute path to a file holding additional decrypt-only master keys, one key per line, blank lines ignored. When set it takes precedence over `additional_master_keys`. |

The master key is key material: supply a high-entropy value generated by a CSPRNG, not a memorable passphrase. Unlike the CryptFs passphrase and the S3 SSE-C key, it is read from the configuration at startup and its strength cannot be validated, so the requirement is on you to provide. It is never stored alongside the ciphertext, so a weak master key is exposed to offline brute-force if the data store is compromised. To change the master key later, keep the current key in the configuration, re-encrypt the stored secrets onto the new key with the [`convertsecrets`](cli.md#re-encrypting-the-stored-secrets) command, and only then update the configuration. Simply replacing the key in the configuration without running `convertsecrets` first makes the secrets encrypted with the previous key unrecoverable.

#### Master key ring

`master_key`/`master_key_path` hold the single **active** key, used to encrypt and tried first on decryption. `additional_master_keys` (and the file referenced by `additional_master_keys_path`) hold **decrypt-only** keys, tried in order only when the active key fails to decrypt a secret. This lets the data store hold secrets written under more than one key at the same time, which is what makes a master key rotation possible without stopping the server on a networked SQL database (PostgreSQL, MySQL/MariaDB, CockroachDB), whether a single instance or a cluster: the running instances can decrypt both the old and the new key while the secrets are migrated. With no additional keys the configuration is a plain single-key setup.

```json
{
  "kms": {
    "secrets": {
      "master_key": "<active-key>",
      "additional_master_keys": ["<previous-key>"]
    }
  }
}
```

The additional keys can also be kept in a file, one key per line (blank lines are ignored). When `additional_master_keys_path` is set it **takes precedence** over `additional_master_keys`, which is then ignored — the file is the single source of the decrypt-only keys, mirroring how `master_key_path` overrides `master_key` for the active key:

```json
{
  "kms": {
    "secrets": {
      "master_key": "<active-key>",
      "additional_master_keys_path": "/etc/sftpgo/additional_master_keys"
    }
  }
}
```

Additional keys require an active key: they are decrypt-only fallbacks for the key being rotated, so setting `additional_master_keys`/`additional_master_keys_path` while `master_key`/`master_key_path` is empty is a misconfiguration and is rejected at startup.

Each additional key is trimmed of surrounding whitespace and empty or duplicate keys (including one equal to the active key) are dropped. When `additional_master_keys` is set through an environment variable (`SFTPGO_KMS__SECRETS__ADDITIONAL_MASTER_KEYS`) the values are comma separated, so a key containing a comma must be provided via the configuration file or `additional_master_keys_path` instead; hex or base64 / base64url encoded keys (CSPRNG values always are) never contain a comma. At startup the log reports how many keys the ring holds, so you can confirm the configuration was parsed as intended.

See [rotating the master key without downtime](cli.md#rotating-the-master-key-without-downtime) for the full procedure.

You can define multiple configurations (e.g., one with `sha256` or `sha512` and another with `sha1`) and instruct users to pick the appropriate one. The algorithm should not be changed if there are users or admins using the configuration. |

## SMTP

SMTP configuration enables SFTPGo email sending capabilities.

Configuration parameters for the `smtp` section.

| Parameter | Type | Default | Description |
| ----------- | ------ | --------- | ------------- |
| `host` | string | empty | SMTP server hostname. Leave empty to disable email sending capabilities. |
| `port` | integer | `587` | SMTP server port. |
| `from` | string | empty | From address, for example `SFTPGo <sftpgo@example.com>`. Many SMTP servers reject emails without a `From` header; if not set, SFTPGo will try to use the username as a fallback, which may or may not be appropriate. |
| `user` | string | empty | SMTP username. |
| `password` | string | empty | SMTP password. Leaving both username and password empty disables SMTP authentication. |
| `auth_type` | integer | `0` | Authentication type. `0` = Plain, `1` = Login, `2` = CRAM-MD5, `3` = XOAUTH2. |
| `encryption` | integer | `0` | Encryption mode. `0` = none, `1` = TLS, `2` = STARTTLS. |
| `domain` | string | empty | Domain to use for the `HELO` command. If empty, `localhost` is used. |
| `templates_path` | string | empty | Path to email templates. Can be absolute or relative to the config dir. Templates are searched within a subdirectory named `email` in the specified path. You can customize email templates by specifying an alternate path and placing your custom templates there. |
| `debug` | integer | `0` | Set to `1` to enable SMTP debug logging. |
| `oauth2` | struct | | OAuth2-related configuration. See [SMTP OAuth2](#smtp-oauth2) below. |

## Plugins

:warning: The plugin system is experimental. Configuration parameters and interfaces may change in a backward-incompatible way in future releases.

The `plugins` section defines a list of external plugins. Each plugin is configured using a struct with the following fields:

| Parameter | Type | Default | Description |
| ----------- | ------ | --------- | ------------- |
| `type` | string | | Plugin type. Supported types: `notifier`, `kms`, `auth`, `eventsearcher`, `ipfilter`, `syncer`. |
| `notifier_options` | struct | | Options for notifier plugins. See [notifier options](#notifier-options) below. |
| `syncer_options` | struct | | Options for syncer plugins. See [syncer options](#syncer-options) below. |
| `kms_options` | struct | | Options for KMS plugins. See [KMS options](#kms-plugin-options) below. |
| `auth_options` | struct | | Options for auth plugins. See [auth options](#auth-plugin-options) below. |
| `cmd` | string | | Path to the plugin executable. |
| `args` | list of strings | empty | Optional arguments to pass to the plugin executable. |
| `sha256sum` | string | empty | SHA256 checksum for the plugin executable. If not empty, it is used to verify the integrity of the executable. |
| `auto_mtls` | boolean | `false` | If enabled, the client and server automatically negotiate mutual TLS for transport authentication. This ensures that only the original client is allowed to connect to the server (all other connections are rejected), and the client refuses to connect to any server that is not the original instance started by the client. |
| `env_prefix` | string | empty | Prefix for environment variables to pass from the SFTPGo process environment to the plugin. Set to `none` to pass no environment variables, or `*` to pass all. If empty, the prefix is derived from the plugin name in uppercase with `-` replaced by `_` and a trailing `_`. For example, if the plugin name is `sftpgo-plugin-eventsearch`, the prefix will be `SFTPGO_PLUGIN_EVENTSEARCH_`. |
| `env_vars` | list of strings | empty | Additional environment variable names to pass from the SFTPGo process environment to the plugin. |

### Notifier options

| Parameter | Type | Default | Description |
| ----------- | ------ | --------- | ------------- |
| `fs_events` | list of strings | empty | Filesystem events to notify. Supported values: `download`, `upload`, `first-upload`, `first-download`, `delete`, `rename`, `mkdir`, `rmdir`, `copy`, `ssh_cmd`. |
| `provider_events` | list of strings | empty | Provider events to notify. Supported values: `add`, `update`, `delete`. |
| `provider_objects` | list of strings | empty | Provider objects to notify. Supported values: `user`, `folder`, `group`, `admin`, `api_key`, `share`, `event_action`, `event_rule`, `role`, `ip_list_entry`, `configs`, `license`. |
| `log_events` | list of integers | empty | Log events to notify. `1` = Login failed, `2` = Login with non-existent user, `3` = No login tried, `4` = Algorithm negotiation failed, `5` = Login succeeded, `6` = Share legal agreement accepted. |
| `retry_max_time` | integer | `0` | Maximum number of seconds an event can be late. SFTPGo timestamps each event and queues any events that the plugin fails to handle. Events older than this limit are discarded. SFTPGo retries queued events every 30 seconds. `0` means no retry. |
| `retry_queue_max_size` | integer | `0` | Maximum number of events the internal queue can hold. Once the queue is full, events that cannot be sent to the plugin are discarded. `0` means no limit. |
